# Security Policy


## Supported Versions

We actively maintain and patch the following versions of VolunteerIQ:

| Version | Supported |
|---|---|
| `main` branch (latest) | ✅ Yes |
| Older tagged releases | ❌ No — please upgrade |

---

## Reporting a Vulnerability

**Please do NOT open a public GitHub issue for security vulnerabilities.** Public disclosure before a fix is in place puts users and NGO data at risk.

### How to Report

Send a detailed report to:

**📧 Email:** `security@deepcraft.dev` *(replace with your actual security contact)*

Or use [GitHub's private vulnerability reporting](https://docs.github.com/en/code-security/security-advisories/guidance-on-reporting-and-writing/privately-reporting-a-security-vulnerability) on this repository.

### What to Include

Please provide as much of the following information as possible:

- **Type of vulnerability** (e.g., Firestore rule bypass, API key exposure, IDOR, XSS, etc.)
- **Affected component** (API route, client page, Firestore rule, etc.)
- **Steps to reproduce** — clear, step-by-step instructions
- **Proof of concept** — code snippet, curl command, or screenshots (no need to exploit live data)
- **Impact assessment** — what data or users are at risk
- **Suggested fix** (optional but appreciated)

---

## Response Timeline

| Action | Timeline |
|---|---|
| Acknowledgement of report | Within **48 hours** |
| Initial assessment and severity triage | Within **5 business days** |
| Fix deployed to `main` | Within **14 days** for critical, **30 days** for moderate |
| Public disclosure (coordinated) | After fix is deployed + 7-day grace period |

We follow [coordinated disclosure](https://cheatsheetseries.owasp.org/cheatsheets/Vulnerability_Disclosure_Cheat_Sheet.html). We will credit researchers who report valid vulnerabilities (unless they prefer to remain anonymous).

## Known Security Design Decisions

The following design decisions are intentional and are **not** vulnerabilities:

### 1. Firebase Client Config is Public
`NEXT_PUBLIC_FIREBASE_*` variables are exposed to the browser by design — this is standard Firebase practice. The Firebase Security Rules are the true access control layer, not the client config. Our Firestore rules ensure:
- Volunteers can only read/write their own `/users/{uid}` document
- Only coordinators can update `/issues` status
- No unauthenticated reads on any collection

### 2. Cookie-Based Session (non-HttpOnly)
We store `viq_uid` and `viq_role` as browser cookies for middleware-level route protection. These are supplemental guards — the actual data access is controlled by Firebase Auth tokens on the server side (Firebase Admin SDK verifies identity server-side in API routes).

### 3. WhatsApp Links are wa.me Deep Links
The `wa.me` links generated by VolunteerIQ are standard WhatsApp deep links that pre-fill a message. No credentials or sensitive tokens are embedded in these links. They are generated server-side and sent to coordinators only.

### 4. Gemini API Key Rotation
The Gemini API keys (`GEMINI_API_KEY`, `GEMINI_API_KEY_2`, etc.) are stored only in `.env.local` and are **never** exposed to the client. All Gemini calls are made from Next.js API routes (server-side).

## Security Best Practices for Deployers

If you are self-hosting or forking VolunteerIQ, please ensure:

- [ ] `.env.local` is in `.gitignore` and never committed
- [ ] `FIREBASE_ADMIN_PRIVATE_KEY` is stored as a secret in your deployment platform (Vercel Environment Variables)
- [ ] Firebase Security Rules are deployed (see `docs/SETUP.md`)
- [ ] Firebase Auth has email enumeration protection enabled
- [ ] The Firestore database is **not** in test mode (open read/write)
- [ ] API routes are not publicly accessible without authentication where applicable

---

## Scope

| In Scope | Out of Scope |
|---|---|
| Firestore data access control bypasses | Third-party services (Firebase, Vercel, Google infrastructure) |
| API route authentication bypasses | Volunteer's own device security |
| XSS vulnerabilities in rendered issue content | Brute-force attacks on Firebase Auth (handled by Google) |
| Exposure of admin private keys | Social engineering |
| IDOR — accessing another user's profile data | Denial of service |
